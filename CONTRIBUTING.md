# Contributing to NovaVision

Thanks for your interest. NovaVision is a small, reproducible research artifact: a protocol and evaluation harness for emotion-controllable text-to-image generation, built around the failure modes that make such a measurement easy to fake. Contributions that improve rigor, reproducibility, or clarity are especially welcome.

## Development setup

Everything is driven by the Makefile from the repository root:

```bash
make setup      # core + dev deps (deterministic core, tests, lint)
make test       # run the pytest suite, no models needed (runs in seconds)
make lint       # ruff check + format
make setup-ml   # model runtime (torch, transformers, diffusers, CLIP)
make pilot      # the committed CPU pilot (256-px, 2 subjects, 1 seed -> n=14)
make paper      # regenerate the paper tables from results/paper/results.json
```

The pilot results (results/paper/results.json) and AffectBench are committed, so tests and the paper tables run without downloading any models. make setup-ml then make pilot rebuilds the committed CPU pilot; make reproduce runs the powered content-track (512-px, 3 seeds) and needs a GPU box.

Python 3.9 to 3.12 are supported and all tested in CI.

## Before opening a pull request

Run the full local gate; all should be clean:

```bash
make test
ruff check novavision tests scripts server.py app.py
ruff format --check novavision tests scripts server.py app.py
mypy novavision --ignore-missing-imports
```

CI also runs a coverage gate, a secret scan (gitleaks), and a dependency audit (pip-audit) on Python 3.9 to 3.12.

Then:

- The suite is **166 tests**; add or update one for any behavior change. `tests/test_contracts.py`
  pins cross-module invariants (label ordering, backend-default parity, seed pairing, provenance
  API) that per-module tests miss; extend it when you add a coupling across modules. Statistics live in novavision/eval/metrics.py and are unit-tested to the edge cases (nan, ties, small-n).
- Keep the deterministic core dependency-light: heavy deps (torch, transformers, diffusers, CLIP) are imported *inside* the functions that use them, never at module top.
- If you changed the pilot, commit the regenerated results.json and run make paper so the tables stay in sync.

## Style

- Clean code, few comments: let names and structure carry the meaning.
- Keep functions short and single-purpose; one logical change per commit.
- Commit messages are short and human in the "updated X" convention: one or two words describing what changed (e.g. "updated paper", "trim paper", "cross-split dedup", "honest framing").

## Scientific changes

The contribution is the protocol and its honesty, not a single controllability score. Please keep these integrity rules intact:

- **Never let a reported number come from a non-reproducible path.** load_benchmark has no default and the null backend is for tests only; the test fixture can never back a result. results.json logs a full provenance manifest (git SHA, library versions, model and dataset revisions, benchmark hash), and tables and figures are regenerated by scripts, never hand-written.
- **Don't weaken the controls.** The two floors (raw, the no-emotion negative control at chance 1/7; scene, the template-only positive control and ceiling), the shuffled-label permutation control (n=2000), and the probe-collapse diagnostic (probe_health) exist to keep the headline honest. The committed result is an honest null precisely because they all bind: extend these guards, don't bypass them.
- **Keep model and data licenses straight** (see [THIRD_PARTY_LICENSES.md](THIRD_PARTY_LICENSES.md)); the repo's MIT covers code only, not the bundled models (SD-Turbo, CLIP, the emotion classifier) or the GoEmotions/EmoSet-derived data.
